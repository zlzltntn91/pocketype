# 프로젝트 가이드: tutorial-hannes

## 역할
기본은 **직접 처리**다. 서브에이전트(`tester`, `implementer`, `reviewer`, `refactorer`)를 동원하는 멀티에이전트 TDD 플로우는 사용자가 `/feature` 슬래시 명령으로 명시적으로 요청했을 때만 실행한다.
서브에이전트 정의는 `.claude/agents/` 참고.

## 기본 처리 지침
1. 요구사항 모호 시 `AskUserQuestion` 필수. 추측 금지.
2. 변경 후 해당 스택의 **테스트 + 린트**를 직접 실행해 통과 확인 후 보고.
3. 실패하면 같은 프로세스 안에서 바로 고친다.

## `/feature` 권장 요청 유형
다음 종류의 작업은 품질 게이트가 더 엄격해야 하므로 `/feature`로 요청하는 것을 권장한다. 사용자가 그냥 평문으로 요청하면 오케스트레이터는 **한 번** `/feature로 진행할지` 확인한 뒤, 아니라고 하면 직접 처리한다.
- 새 기능 추가 (새 라우트/페이지/엔드포인트/도메인 함수).
- 계산 규칙·비즈니스 로직 변경 (예: `rankLeaders` 산정식, 점수 공식).
- 데이터 스키마/타입 계약 변경 (`TypeInfo` 필드 추가 등).
- 여러 파일에 걸친 구조 리팩터 (파일 이동, 모듈 경계 재설계).
- 보안/권한/영속성에 영향을 주는 변경.

## 직접 처리 예시
- 카드 뱃지 12개를 2개로 합쳐 라벨만 재배치 (표시 변경).
- 버튼 색상 수정 + 호버 클래스 추가.
- 타이포/리네임/1~2줄 수정, 주석/문서만 변경.
- 기존 테스트/린트/빌드 실행만 하고 보고.
- 단일 파일 내 심볼 리네임/추출 (공개 API 불변).

## 품질 게이트 명령 (frontend 기준)
- 테스트: `npm test`
- 린트: `npm run lint`
- 빌드/타입체크: `npm run build`
- 다른 스택은 각 프로젝트의 동등 명령으로 치환 (Java: `./gradlew test`, `./gradlew check`, `./gradlew build` 등).

## TDD 플로우 (`/feature` 전용, Red → Green → Review → Refactor)

1. **요구사항 명확화**
   - 모호하면 `AskUserQuestion`으로 사용자에게 질문한다. 추측하지 않는다.

2. **Red** — `Agent(subagent_type="tester")` 호출
   - 프롬프트에 포함: 기능 요구사항, 대상 파일/모듈 경로, 테스트 실행 명령.
   - tester는 실패하는 테스트를 작성하고 실행해 실패를 확인한 뒤 테스트 파일 경로/실행 명령/실패 로그를 반환한다.

3. **Green** — `Agent(subagent_type="implementer")` 호출
   - 프롬프트에 포함: 2에서 받은 테스트 파일 경로, 테스트 실행 명령, **린트 실행 명령**, 테스트가 통과해야 하는 동작 설명.
   - implementer는 테스트를 통과시키는 최소 구현을 작성하고, **테스트 + 린트 둘 다 통과한 로그**를 반환한다.

4. **Review** — `Agent(subagent_type="reviewer")` 호출
   - 프롬프트에 포함: 변경된 파일 경로 전체 목록, 테스트 실행 명령, 린트 실행 명령.
   - reviewer는 `승인: YES` 또는 `승인: NO` + 지적사항 목록을 반환한다.
   - `YES`면 **최종 검증 단계(7)로 진행**.

5. **Refactor** — `Agent(subagent_type="refactorer")` 호출
   - 프롬프트에 포함: reviewer가 반환한 지적사항 전체, 테스트 실행 명령, 린트 실행 명령.
   - refactorer가 개선 후 **테스트 + 린트 둘 다 통과**한 로그를 반환하면 4로 복귀한다.

6. 4 ↔ 5 루프는 **최대 3회**. 3회 후에도 `승인: NO`면 현황(잔여 지적사항, 마지막 테스트/린트 결과)을 사용자에게 보고하고 종료한다.

7. **최종 검증 (오케스트레이터 직접 수행)**
   - 사용자에게 보고하기 전에 오케스트레이터가 직접 `npm run build` (또는 프로젝트의 동등한 빌드/타입체크 명령)를 실행해 타입 에러가 없는지 확인한다.
   - 실패하면 에러 로그와 함께 refactorer를 다시 호출(루프 카운트 외)하고, 통과하면 사용자에게 최종 보고.

## 서브에이전트 호출 규칙 (`/feature` 내부 전용)
- 서브에이전트는 부모 컨텍스트를 상속하지 않는다 (콜드 스타트). 호출 프롬프트에 **파일 절대 경로, 테스트 실행 명령, 직전 단계 결과**를 반드시 명시한다.
- 한 호출에서 하나의 명확한 목표만 전달한다.

## 학습 누적
세션 종료 시 `.claude/learnings.md` 업데이트는 Stop hook이 자동으로 수행한다. 오케스트레이터가 별도로 처리하지 않는다.
